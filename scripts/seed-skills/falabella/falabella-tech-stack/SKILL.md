# forge tech stack + code quality gates

This skill answers two questions every team asks: "what can we use?" and
"what does our PR have to pass?". The approved stack lives behind the
platform's SSO; this skill bundles a synced snapshot for offline use and
explains the exception process when the answer is "not yet".

## Where the source of truth lives

The canonical stack is the JSON at
`https://platform.forge.internal/tech-stack/v1.json`, behind the
forge SSO (Keycloak). Authenticated developers can fetch it directly:

```
curl -H "Authorization: Bearer ${forge_PORTAL_TOKEN}" \
  https://platform.forge.internal/tech-stack/v1.json
```

The token is the same one issued by the FDH portal — log in to
`https://fdh.forge.internal`, copy your access token from the
profile page, and use it for `Authorization: Bearer`.

A synced snapshot sits in `references/tech-stack.json` next to this
skill. It is regenerated by the FDH publish pipeline whenever the
canonical JSON changes. Read the snapshot when the network call is
inconvenient; trust the live URL when stakes are high.

## How to use this skill

1. **Identify the stack categories that apply to the change.** A new
   service touches several (language, framework, database, queue,
   observability). A library touches few. A config change may touch
   none.
2. **Check each chosen technology against the matching `approved` list.**
   Anything in `approved` ships without further approval.
3. **Check the `evaluating` list.** Items there have a sponsor team
   and an open conversation. You can use them with the sponsor's
   sign-off; reflect that decision in `design.md` of your change.
4. **Anything not in either list → exception process** (below).

## The shape of the JSON

```
{
  "schema_version": 1,
  "generated_at": "ISO 8601",
  "languages":   { "approved": [...], "evaluating": [...], "deprecated": [...] },
  "frameworks":  { "<language>": { "approved": [...], ... } },
  "databases":   { "approved": [...], "evaluating": [...], "deprecated": [...] },
  "queues":      { "approved": [...], ... },
  "observability": { "approved": [...], ... },
  "code_quality": {
    "<language>": {
      "linter":     { "name": "...", "config_url": "..." },
      "formatter":  { "name": "...", "config_url": "..." },
      "test_runner":{ "name": "...", "min_coverage": 70 },
      "sonarlint":  { "ruleset_url": "...", "min_quality_gate": "passing" }
    }
  },
  "exception_process_url": "..."
}
```

See `references/tech-stack.json` for the current shape with sample
values.

## Code quality gates every service runs

Per language, the JSON's `code_quality` block names four gates that
every PR must pass before merge:

- **linter** — language-native linter with the forge ruleset.
- **formatter** — auto-formatter; CI fails on diff.
- **test_runner** — language-native test runner with the minimum
  coverage declared in the JSON (`min_coverage`).
- **sonarlint** — SonarLint against the central ruleset; CI gates on
  `min_quality_gate` (`passing`, `warning`, or `none`).

Common per-language combinations (see the JSON for the live answer):

| Language | Linter | Formatter | Test runner | SonarLint |
|---|---|---|---|---|
| Go | `golangci-lint` | `gofmt -s` | `go test -race -cover` | `passing` |
| TypeScript | `eslint` (next/core-web-vitals) | `prettier` | `vitest` or framework default | `passing` |
| Java | `checkstyle` + `spotbugs` | `spotless` | `JUnit 5` | `passing` |
| Python | `ruff` | `ruff format` | `pytest --cov` | `passing` |
| Kotlin | `ktlint` + `detekt` | `ktlint -F` | `JUnit 5` / `Kotest` | `passing` |

## How to add a new technology

1. Open a change in the OpenSpec hub (`/opsx:propose tech-add-<thing>`).
2. The proposal explains WHY this technology is needed and what
   approved option falls short.
3. The design lists evaluated alternatives + chosen one with rationale.
4. Sponsor team commits to a six-month adoption review.
5. After review, the technology either graduates to `approved` or is
   removed from `evaluating`.

The full exception process is at the URL declared by the JSON's
`exception_process_url` field.

## Anti-patterns

- **Picking a tech without reading the JSON.** "Everyone uses X so it
  must be fine" is the most common cause of stack drift.
- **Adding a linter or test framework that isn't in the per-language
  `code_quality` block.** Maintenance cost falls on whoever picked it.
- **Disabling SonarLint rules to make the gate pass.** Open a change to
  modify the ruleset instead; rule changes go through the platform team.
- **Forking the JSON for "our team's stack".** The whole point of a
  single source of truth is that it's single. Open a change to modify it.

## How this skill stays fresh

The `references/tech-stack.json` snapshot is regenerated by the FDH
publish pipeline. When the canonical URL's `generated_at` timestamp is
newer than the snapshot's, CI republishes this skill with the updated
JSON. The skill's own `metadata.last_synced` field reflects the snapshot
version, not necessarily the live JSON.
